<objective>
Disparar `git fetch --prune --quiet` automaticamente ao entrar em `select_branch`, com timeout hard de 5 segundos. Indicador visual durante a espera. Falhas degradam graciosamente (lista local stale + mensagem informativa).

**Purpose:** Hoje o usuário precisa sair do launcher, rodar `git fetch`, e voltar pra ver branches novas/atualizadas no remote. Como o workflow é "abre repos on-demand para iniciar trabalho", fetch just-in-time é o ponto natural.

**Output:** wrapper `fetch_with_timeout` + integração no início de `select_branch`.
</objective>

## T3.2 — Implementar `fetch_with_timeout`

**Onde:** `repo-launcher.sh` — adicionar função antes de `select_branch` (~linha 675).

**Signature:** `fetch_with_timeout <repo_path>` → exit 0 (success), 1 (git error), 124 (timeout)

**Implementação preferencial (com `timeout`):**

```bash
fetch_with_timeout() {
  local repo_path="$1"
  if ((HAS_TIMEOUT)); then
    timeout 5s bash -c '(cd "$1" && git fetch --prune --quiet)' _ "$repo_path" 2>/dev/null
    return $?
  fi
  # fallback: bash puro
  _fetch_bash_timeout "$repo_path"
}
```

**Fallback bash puro:**

```bash
_fetch_bash_timeout() {
  local repo_path="$1"
  local timeout_sec=5
  local poll_interval=0.1
  local max_iterations=$((timeout_sec * 10))   # 50 iterations × 0.1s = 5s

  (cd "$repo_path" && git fetch --prune --quiet 2>/dev/null) &
  local pid=$!

  local elapsed=0
  while kill -0 "$pid" 2>/dev/null; do
    if ((elapsed >= max_iterations)); then
      kill -9 "$pid" 2>/dev/null
      wait "$pid" 2>/dev/null
      return 124
    fi
    sleep "$poll_interval"
    elapsed=$((elapsed + 1))
  done

  wait "$pid"
  return $?
}
```

**Por quê `bash -c '... "$1"' _ "$repo_path"`:** evita interpolação direta do path no shell — defesa contra paths com espaços ou caracteres especiais.

select_branch() {
  local repo="$1"
  local repo_path="$GITHUB_DIR/$repo"
  SELECTED_WORKTREE_PATH=""

  # Auto-fetch just-in-time (timeout 5s, fail-soft)
  local c_green=$'\033[38;2;166;227;161m'

<risks>
- **`timeout` pode não estar no PATH:** detectado em T3.1, fallback bash assume.
- **Bash kill em git-bash:** `kill -9` em git child no MSYS2 às vezes não funciona perfeitamente. Validar T3.6 explicitamente.
- **Bloqueio de UX por 5s:** intencional. Mas se o user achar incômodo, alternativa é fetch async em background — fora do escopo desta wave.
- **Race condition no PID:** após `kill -9 $pid`, há janela curta onde PID pode ser reciclado. Mitigado pelo `wait $pid` imediato após kill.
- **Auth interativo:** se o repo tem remote SSH sem agent rodando, fetch pode prompt por passphrase e ficar congelado. Timeout pega isso.
</risks>
